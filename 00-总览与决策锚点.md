## 1. 决策锚点 D1–D6

### D1 · CSR 首屏「零 JS 可见」= A + B 并存

**A 路径**：Vite/webpack 构建期把 `snippet.html` 注入 `index.html`（**page 级**）

- 因为 SPA 是单 `index.html`，多路由必须由 client-side bridge 在 hydration 前根据 `location.pathname` **换骨架**
- 详见 [11-step2-vite-plugin-SSG-lite.md](./11-step2-vite-plugin-SSG-lite.md) + [12-step3-SPA-router-bridge.md](./12-step3-SPA-router-bridge.md)

**B 路径**：SWC 把 `bones.json` 内联到对应 chunk，`<Skeleton>` 挂载瞬间渲染（**组件级**）

- 兼容现有 `<Skeleton name="x" loading>` 用法
- 详见 [13-step4-SWC-runtime-inject.md](./13-step4-SWC-runtime-inject.md)

A 解决"首屏白屏到看见骨架的间隔"，B 解决"路由内任意组件的接口态/局部骨架"。两者**不可互相替代**。

### D2 · 三端模型 = B1（共享 schema + 各端独立采集）+ 按"类型 × 平台"分目录（v2 调整）

```
bones/
  pages/                     # page 级骨架（原 ssr，更名为业务术语）
    web/{name}.bones.json + {name}.snippet.html + manifest.json
    rn/{name}.bones.json
    mp/{name}.bones.json + pages/{name}.wxml.partial
  regions/                   # region 级骨架（原 api，更名为业务术语）
    web/{region}.bones.json + bindings.json
    rn/{region}.bones.json
    mp/{region}.bones.json
```

变化（vs v1）：

- `ssr/api` 抬到一级，命名换为业务术语 `pages/regions`（无歧义、一眼区分）
- 平台变成二级，扫一眼就知道当前看的是哪种产物
- CI artifact 上传 `bones/regions/**` vs `bones/pages/**` 更直观

不变：

- **三端 `bones.json` schema 同形**（[01-架构与模型.md §3](./01-架构与模型.md)），但 `platform` 字段区分
- **各端独立采集**：Web 用 BGv2（[02 §1](./02-最佳生成算法.md)）；RN 用 `measure()`；MP 用 Taro H5 预览态 + BGv2
- 不做"Web 采一份映射到 RN/MP"——准度太低

### D3 · 严格深层依赖 = esbuild metafile + 深度 5 + ignore glob

`smarty check` 重算 hash 的输入：

```
inputs = ∪{
  传递依赖深度 ≤ 5 的所有源文件，去掉匹配 ignoreGlobs 的，
  从 esbuild build({ metafile: true }) 的 metafile.inputs 拿到
}
hash = sha256(sorted(file → content))
```

- **不用 mtime**（[ssr-injection-design.md 决议 D 已证不可靠](../boneyard-main/packages/boneyard/src/ssr-injection-design.md)）
- **不用 madge / dependency-tree**（不解析 dynamic import）
- **不自写 AST walker**（重）
- **默认 ignoreGlobs**：`['**/design-tokens.*', '**/icons/**', '**/node_modules/**', '**/*.svg', '**/*.css']`，可配
- **深度 5**：默认上限；防止图爆炸；超过深度的依赖标记为 `_outOfDepth`，仅记不参与 hash
- 详见 [18-step9-check-CLI-深层依赖.md](./18-step9-check-CLI-深层依赖.md)

### D4 · 捕获 = Playwright 权威 + DevSave 仅人工调试

**职责重新分工**（v2 调整）：

| 通路 | 触发 | 产物落点 | 职责 |
|---|---|---|---|
| **Playwright 批量** | `smarty build` / CI 定时 / pre-push | **`bones/`（进 git，权威源）** | 唯一进 CI hash 校验的产物；含 `*.bones.json` + `{name}@{w}.fixture.png` + `{name}@{w}.skeleton.png` + `{name}@{w}.diff.png`（若超阈值） |
| **DevSave** | `dev:ske` 开发者主动浏览 | **`.smarty-cache/`（不进 git）** | 仅供开发者 debug / 预览效果；**不参与 check hash**；CI 完全忽略 |

要点：

- DevSave 不再是"权威"；它就是个**便利工具**。开发者写完组件想立刻看骨架长啥样 → 浏览页面 → `.smarty-cache/{name}.preview.bones.json` 出现 → HMR 业务侧用这份预览
- 真正进 git、被 CI 比对的，**只有** Playwright 跑出来的 `bones/`
- 这避免了"开发者只在 1280px 浏览过、375px 漏采"的灾难
- **Visual Diff 是新增能力**（11 个调研项目都没做）：用 `pixelmatch` 算 fixture 渲染 vs skeleton 渲染的像素差
- 默认 diff 阈值 5%（可配），超限 → CI fail + PR 注释
- 详见 [17-step8-Playwright批量与Visual-Diff.md](./17-step8-Playwright批量与Visual-Diff.md) + [16-step7-DevSave-与dev-ske.md](./16-step7-DevSave-与dev-ske.md)

### D5 · SSG-lite 注入位置 = `index.html` 的 `head`/`body`/`auto` 三档

原方案 `inject: 'auto'|'head'|'body'` 三档语义保留，但**作用对象从 SSR 响应流改为构建期 `index.html`**：

- `body`（默认，最佳 FP）：注入到 `</body>` 前
- `head`：仅注入 `<style>` + `<script>`，overlay HTML 由 boot 时 `createElement` 挂到 body（详见 [11-step2 §3](./11-step2-vite-plugin-SSG-lite.md)）
- `auto`：能注 body 就注 body，否则降级 head

原 Transform Stream 中间件、`Content-Length` 处理、Edge Runtime 适配 → [42-Open-Questions-后置.md](./42-Open-Questions-后置.md) 后置（CSR 项目用不上）。

### D6 · 文档落点 = 顶层独立目录 `/重构方案/`

- 旧 3 份设计文档（`packages/boneyard/src/skeleton-architecture-design.md` / `ssr-injection-design.md` / `skeleton-build-pipeline-design.md`）作为历史归档**不动**
- 新方案的实施目标在 `/重构方案/`，与代码物理隔离便于跨团队评审
- 实施阶段如需引用旧文档某章节，用相对路径链接（如 `../boneyard-main/packages/boneyard/src/...`）

## 3. 与原 3 份设计文档的差异（精简到 1 张表）

| 主题 | 原文档主张 | 本方案 |
|---|---|---|
| 调度器 | 完整 SkeletonScheduler（heap + lane + expiration + host abstraction） | 暂不实现，预留接口；先用 `requestIdleCallback` + `MessageChannel` 直接用 |
| SSR | 中间件 + Transform Stream + `Content-Length` 处理 | CSR 项目无需，改 SSG-lite 注入 `index.html`；中间件留作 Open Questions |
| 接口态 | binding graph + Tier 1/2/3 自动关联 + autoBound | 只做显式 `<Bound deps={['…']}>` |
| 三端 | 同一份 `bones.json` 跨端复用 | schema 同形 + 各端独立采集 + 按平台分目录 |
| check | mtime 预筛 + 浅层依赖 | esbuild metafile + 深度 5 + ignore glob 严格内容 hash |
| 生成算法 | 直接复用 `renderBones()`，靠 fixture + DOM | **BGv2 v2 算法**（融合 11 项目精华，[02](./02-最佳生成算法.md)） |
| 注入位置 | SSR 响应流的 head/body | SSG-lite 构建期写入 `index.html` 的 head/body |
| 文档 | 散在 packages/boneyard/src/ | 顶层 `/重构方案/` 18 份文档统一组织 |

## 4. 决策修改流程

任何想推翻 D1–D6 的提议，需在 PR 描述里包含：

1. **触发场景**：什么实测数据让原决策不再合理
2. **影响面**：哪些 step 文档需要重写、哪些已写代码需要回滚
3. **替代方案**：新决策的具体形式（不是"我们应该重新考虑"，而是"D1 改为 X"）
4. **PR 同步修改本文** D1–D6 表格 + [42-Open-Questions-后置.md](./42-Open-Questions-后置.md) 的对应条目
